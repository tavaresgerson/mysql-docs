## 8.7 Otimizando para Tabelas MEMORY

Considere usar tabelas `MEMORY` para dados não críticos que são acessados com frequência, e que são somente leitura (`read-only`) ou raramente atualizados. Execute um `Benchmark` em sua aplicação contra tabelas `InnoDB` ou `MyISAM` equivalentes sob uma carga de trabalho realista, para confirmar se o desempenho adicional justifica o risco de perda de dados, ou a sobrecarga de copiar dados de uma tabela baseada em disco (`disk-based table`) na inicialização da aplicação.

Para obter o melhor desempenho com tabelas `MEMORY`, examine os tipos de `Queries` realizadas em cada tabela e especifique o tipo de `Index` a ser usado para cada um, seja um `B-tree index` ou um `hash index`. Na instrução `CREATE INDEX`, use a cláusula `USING BTREE` ou `USING HASH`. `B-tree indexes` são rápidos para `Queries` que realizam comparações de maior ou menor (greater-than or less-than) através de operadores como `>` ou `BETWEEN`. `Hash indexes` são rápidos apenas para `Queries` que buscam valores únicos através do operador `=`, ou um conjunto restrito de valores através do operador `IN`. Para entender por que `USING BTREE` é frequentemente uma escolha melhor do que o padrão `USING HASH`, consulte a Seção 8.2.1.20, “Evitando `Full Table Scans`”. Para detalhes de implementação dos diferentes tipos de `MEMORY indexes`, consulte a Seção 8.3.8, “Comparação de `B-Tree` e `Hash Indexes`”.
