## 8.7 Otimizando para Tabelas de MEMÓRIA

Considere usar as tabelas `MEMORY` para dados não críticos que são acessados frequentemente e que são apenas de leitura ou raramente atualizados. Compruebe sua aplicação contra tabelas equivalentes `InnoDB` ou `MyISAM` sob uma carga de trabalho realista, para confirmar que qualquer desempenho adicional vale o risco de perder dados ou o custo de copiar dados de uma tabela baseada em disco no início da aplicação.

Para obter o melhor desempenho com as tabelas `MEMORY`, examine os tipos de consultas para cada tabela e especifique o tipo a ser usado para cada índice associado, seja um índice de árvore B ou um índice de hash. Na declaração `CREATE INDEX`, use a cláusula `USING BTREE` ou `USING HASH`. Os índices de árvore B são rápidos para consultas que realizam comparações maior que ou menor que por meio de operadores como `>` ou `BETWEEN`. Os índices de hash são rápidos apenas para consultas que buscam valores únicos por meio do operador `=`, ou um conjunto restrito de valores por meio do operador `IN`. Para saber por que `USING BTREE` é frequentemente uma escolha melhor do que o padrão `USING HASH`, consulte a Seção 8.2.1.20, “Evitar varreduras completas da tabela”. Para detalhes de implementação dos diferentes tipos de índices `MEMORY`, consulte a Seção 8.3.8, “Comparação de índices de árvore B e de hash”.
