#### 16.2.3.2 Monitoramento das Replication Applier Worker Threads

Em uma replica multithreaded, as tabelas do Performance Schema [`replication_applier_status_by_coordinator`](performance-schema-replication-applier-status-by-coordinator-table.html "25.12.11.5 The replication_applier_status_by_coordinator Table") e [`replication_applier_status_by_worker`](performance-schema-replication-applier-status-by-worker-table.html "25.12.11.6 The replication_applier_status_by_worker Table") exibem informações de status para a coordinator thread e as applier worker threads da replica, respectivamente. Para uma replica com múltiplos channels, as threads para cada channel são identificadas.

A coordinator thread de uma replica multithreaded também imprime estatísticas no error log da replica regularmente, caso a configuração de verbosidade esteja definida para exibir mensagens informativas. As estatísticas são impressas dependendo do volume de events que a coordinator thread atribuiu às applier worker threads, com uma frequência máxima de uma vez a cada 120 segundos. A mensagem lista as seguintes estatísticas para o replication channel relevante, ou para o default replication channel (que não é nomeado):

Seconds elapsed : A diferença em segundos entre o tempo atual e a última vez em que esta informação foi impressa no error log.

Events assigned : O número total de events que a coordinator thread enfileirou (queued) para todas as applier worker threads desde que a coordinator thread foi iniciada.

Worker queues filled over overrun level : O número atual de events que estão enfileirados em qualquer uma das applier worker threads excedendo o overrun level, que é definido em 90% do comprimento máximo da queue, de 16384 events. Se este valor for zero, nenhuma applier worker thread está operando no limite superior de sua capacidade.

Waited due to worker queue full : O número de vezes que a coordinator thread teve que esperar para agendar um event porque a queue de uma applier worker thread estava cheia. Se este valor for zero, nenhuma applier worker thread esgotou sua capacidade.

Waited due to the total size : O número de vezes que a coordinator thread teve que esperar para agendar um event porque o limite de [`slave_pending_jobs_size_max`](replication-options-replica.html#sysvar_slave_pending_jobs_size_max) havia sido atingido. Esta system variable define a quantidade máxima de memória (em bytes) disponível para as queues das applier worker threads que contêm events ainda não aplicados. Se um event excepcionalmente grande exceder este tamanho, a transaction é retida até que todas as applier worker threads tenham queues vazias e, então, é processada. Todas as transactions subsequentes são retidas até que a transaction grande seja concluída.

Waited at clock conflicts : O número de nanossegundos que a coordinator thread teve que esperar para agendar um event porque uma transaction da qual o event dependia ainda não havia sido committed (confirmada). Se [`slave_parallel_type`](replication-options-replica.html#sysvar_slave_parallel_type) estiver definido como `DATABASE` (em vez de `LOGICAL_CLOCK`), este valor é sempre zero.

Waited (count) when workers occupied : O número de vezes que a coordinator thread dormiu por um breve período, o que pode ocorrer em duas situações. A primeira situação é quando a coordinator thread atribui um event e descobre que a queue da applier worker thread está preenchida além do underrun level (nível mínimo) de 10% do comprimento máximo da queue, caso em que ela dorme por um máximo de 1 milissegundo. A segunda situação é quando [`slave_parallel_type`](replication-options-replica.html#sysvar_slave_parallel_type) está definido como `LOGICAL_CLOCK` e a coordinator thread precisa atribuir o primeiro event de uma transaction à queue de uma applier worker thread; ela faz isso apenas para uma worker com uma queue vazia, portanto, se nenhuma queue estiver vazia, a coordinator thread dorme até que uma se esvazie.

Waited when workers occupied : O número de nanossegundos que a coordinator thread dormiu enquanto esperava por uma applier worker thread queue vazia (ou seja, na segunda situação descrita acima, onde [`slave_parallel_type`](replication-options-replica.html#sysvar_slave_parallel_type) está definido como `LOGICAL_CLOCK` e o primeiro event de uma transaction precisa ser atribuído).
