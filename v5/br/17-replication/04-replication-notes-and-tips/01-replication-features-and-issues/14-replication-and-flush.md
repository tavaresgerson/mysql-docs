#### 16.4.1.14 Replicação e FLUSH

Algumas formas da instrução [`FLUSH`](flush.html "13.7.6.3 FLUSH Statement") não são registradas (logged) porque poderiam causar problemas se replicadas para uma réplica: [`FLUSH LOGS`](flush.html#flush-logs) e [`FLUSH TABLES WITH READ LOCK`](flush.html#flush-tables-with-read-lock). Para um exemplo de sintaxe, consulte [Seção 13.7.6.3, “FLUSH Statement”](flush.html "13.7.6.3 FLUSH Statement"). As instruções [`FLUSH TABLES`](flush.html#flush-tables), [`ANALYZE TABLE`](analyze-table.html "13.7.2.1 ANALYZE TABLE Statement"), [`OPTIMIZE TABLE`](optimize-table.html "13.7.2.4 OPTIMIZE TABLE Statement") e [`REPAIR TABLE`](repair-table.html "13.7.2.5 REPAIR TABLE Statement") são escritas no binary log e, portanto, replicadas para as réplicas. Isso normalmente não é um problema porque essas instruções não modificam os dados da tabela.

No entanto, este comportamento pode causar dificuldades em certas circunstâncias. Se você replica as privilege tables no `mysql` database e atualiza essas tabelas diretamente sem usar [`GRANT`](grant.html "13.7.1.4 GRANT Statement"), você deve executar um [`FLUSH PRIVILEGES`](flush.html#flush-privileges) nas réplicas para que os novos privilégios entrem em vigor. Além disso, se você usar [`FLUSH TABLES`](flush.html#flush-tables) ao renomear uma tabela `MyISAM` que faz parte de uma tabela `MERGE`, você deve executar [`FLUSH TABLES`](flush.html#flush-tables) manualmente nas réplicas. Essas instruções são escritas no binary log a menos que você especifique `NO_WRITE_TO_BINLOG` ou seu alias `LOCAL`.
