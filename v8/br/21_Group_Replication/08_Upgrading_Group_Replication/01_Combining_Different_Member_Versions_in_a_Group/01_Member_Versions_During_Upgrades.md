#### 20.8.1.1 Versões dos membros durante as atualizações

Durante um procedimento de atualização online, se o grupo estiver no modo de único primário, todos os servidores que não estão atualmente fora de linha para a atualização funcionarão como antes. O grupo elege um novo primário sempre que necessário, seguindo as políticas de eleição descritas na Seção 20.1.3.1, “Modo de Único Primário”. Observe que, se você precisar que o primário permaneça o mesmo durante todo o processo (exceto quando ele mesmo está sendo atualizado), você deve primeiro atualizar todos os segundosários para uma versão maior ou igual à versão do membro primário alvo, e depois atualizar o primário por último. O primário não pode permanecer como primário a menos que esteja executando a versão mais baixa do MySQL Server no grupo. Após a atualização do primário, você pode usar a função `group_replication_set_as_primary()` para reapotá-lo como primário.

Se o grupo estiver no modo multi-primário, menos membros online estarão disponíveis para realizar escritas durante o procedimento de atualização, porque os membros atualizados se juntam no modo de leitura apenas após sua atualização. A partir do MySQL 8.0.17, isso se aplica a atualizações entre versões de patch, e para versões menores, isso se aplica apenas a atualizações entre versões principais. Quando todos os membros tiverem sido atualizados para a mesma versão, a partir do MySQL 8.0.17, eles todos voltam automaticamente para o modo de leitura e escrita. Para versões anteriores, você deve definir `super_read_only` para `OFF` manualmente em cada membro que deve funcionar como primário após a atualização.

Para lidar com uma situação de problema, por exemplo, se você precisar reverter uma atualização ou adicionar capacidade extra a um grupo em uma emergência, é possível permitir que um membro se junte a um grupo online, mesmo que ele esteja executando uma versão do MySQL Server menor do que a versão mais baixa usada pelos outros membros do grupo. A variável do sistema de replicação de grupo `group_replication_allow_local_lower_version_join` pode ser usada nessas situações para substituir as políticas de compatibilidade normais.

Importante

Definir `group_replication_allow_local_lower_version_join` para `ON` *não* torna o novo membro compatível com o grupo; fazer isso permite que ele se junte ao grupo sem quaisquer salvaguardas contra comportamentos incompatíveis dos membros existentes. Portanto, isso deve ser usado com cuidado apenas em situações específicas, e você deve tomar precauções adicionais para evitar que o novo membro falhe devido à atividade normal do grupo. Veja a descrição desta variável para obter mais informações.
